Three Bugs That Taught Me More Than Any Tutorial

Lessons from building Arethos — a mental wellness app where React Native's abstractions broke down and the real engineering started.

Arethos is a mental wellness app I co-built with two teammates for KNUST students. It has an AI-powered journal, a mood tracker with sleep logging, and a reflections tab that surfaces behavioral patterns. We shipped it with React Native and Expo SDK 54, backed by SQLite for fully offline-first storage and a dual-AI fallback system (Groq primary, Anthropic secondary).

The interesting parts aren't in the feature list. They're in the three bugs that forced me to understand what React Native actually does under the hood, instead of just using it.

Bug 1: The Delete Button That Didn't Exist

The journal screen shows recent entries as cards. Each card has a three-dot menu. You tap it, a dropdown appears with a "Delete" option. Simple. Except tapping "Delete" did absolutely nothing.

The dropdown was positioned with zIndex: 10. Visually, it rendered on top of everything. But React Native's zIndex only works between direct siblings in the same parent view. Our dropdown and the card's TouchableOpacity wrapper were not siblings — the dropdown was a child of the card, and the TouchableOpacity wrapping the entire card was intercepting every tap before it could reach the dropdown's buttons.

This is a fundamental difference from CSS. On the web, z-index creates stacking contexts that govern hit-testing globally. In React Native, touch responders are resolved by the view tree's parent-child hierarchy, not by visual layering. A child can never "escape" its parent's touch boundary through zIndex alone.

The fix: Replace the inline dropdown with a Modal bottom-sheet. A Modal renders at the OS level, outside the entire view tree. Nothing can intercept its touches. This is not a workaround — it's the correct pattern for any overlay that needs to receive taps in React Native.

The bottom-sheet shows a preview of the entry, a red "Delete entry" button, and a cancel option. Tapping the dark backdrop dismisses it. It works every time, on every device, because it doesn't fight the framework — it uses the escape hatch the framework provides.

Bug 2: The Slider That Teleported

The mood screen has a sleep slider — drag a knob across a track to log hours slept. Except sometimes, mid-drag, the knob would snap back to position zero. Other times it wouldn't move at all. The behavior was inconsistent and maddening.

The root cause took me a while to find: a stale closure.

PanResponder.create() runs once, inside a useRef, at component mount. Every callback it defines — onPanResponderGrant, onPanResponderMove, onPanResponderRelease — captures the variables in scope at that single moment. Those variables are frozen forever.

My onPanResponderMove was reading a state variable called value (the current sleep level), converting it back to a pixel position, and using that as the drag origin. But value inside the closure was always 0 — its initial state. Every drag calculated its position relative to zero, regardless of where the knob actually was.

// THE BUG: value is always 0 inside the frozen closure onPanResponderMove: (_, gs) => { const startPixel = valueToPixel(value); // value === 0, always const newPos = startPixel + gs.dx; knobX.setValue(newPos); } // THE FIX: save real position into a ref at drag start const dragStartX = useRef(0); onPanResponderGrant: (evt) => { // Capture actual pixel position RIGHT NOW, not from state dragStartX.current = evt.nativeEvent.locationX - KNOB_D / 2; }, onPanResponderMove: (_, gs) => { const newPos = dragStartX.current + gs.dx; // always correct knobX.setValue(Math.max(0, Math.min(newPos, maxWidth))); }

The lesson is broader than React Native. Any system that creates callbacks at initialization time and reuses them later — event listeners in embedded systems, registered callbacks in game engines, hooks in ML training loops — is vulnerable to stale closures. The pattern is always the same: if a callback needs current state, pass it through a mutable reference, not a captured variable.

Bug 3: The Vibration That Took 9 Iterations

This one wasn't a bug. It was a design problem that revealed something about how Android haptics actually work.

We wanted the dark/light theme toggle to produce a specific haptic sensation — a subtle shake in the palm, like the Samsung S23's "no face detected" vibration. Not a harsh buzz. Not a single tap. A gentle multi-pulse tremor.

My first instinct was expo-haptics, the standard Expo library for haptic feedback. It offers presets like ImpactFeedbackStyle.Light, .Medium, .Heavy, and notification types like Success and Error.

On Android, none of them worked. Every preset felt like the same single electronic buzz, just at slightly different intensities. Here's why: expo-haptics calls VibrationEffect.createOneShot(duration, amplitude) under the hood. One shot. One pulse. You can adjust how hard and how long, but it's always a single impulse.

To get a multi-pulse waveform — the kind that feels organic and physical — you need VibrationEffect.createWaveform(), which React Native's built-in Vibration.vibrate([pattern]) calls. The array alternates between wait and vibrate durations in milliseconds.

#ApproachResult
1impactAsync(Medium)Too strong, single buzz
2selectionAsync() × 2, 55ms gapDoesn't feel natural
3impactAsync(Light) × 3Still mechanical
4notificationAsync(Error)"That's the wrong password one"
5Vibration.vibrate([0,40,30,40,30,40])Getting there, too strong
6Vibration.vibrate([0,20,30,20,30,20])Better, still heavy
7Vibration.vibrate([0,12,32,12,32,12])Confirmed — subtle palm shake

The key insight: pulse duration controls perceived intensity, not amplitude. A 40ms vibrate pulse feels harsh regardless of what amplitude the OS chooses. A 12ms pulse barely registers consciously — you feel it in your palm without it interrupting your attention. Three 12ms pulses with 32ms breathing room between them produce the exact sensation we wanted.

We shipped three haptic functions: subtleShake() for mode changes, softTap() for buttons, and successPulse() for save confirmations. The platform-specific branching is isolated in a single 15-line file.

// src/lib/haptics.js — the entire haptics layer export function subtleShake() { if (Platform.OS === "android") { Vibration.vibrate([0, 12, 32, 12, 32, 12]); } else { Haptics.impactAsync(Haptics.ImpactFeedbackStyle.Light); } }

What These Three Bugs Have in Common

Each one forced me past the abstraction layer. React Native presents a clean API — zIndex for layering, state for data, expo-haptics for vibrations — but the abstraction leaks in predictable ways when you push it.

The delete button taught me that touch resolution in React Native follows the view tree, not visual stacking. The slider taught me that closures in hook-based architectures can freeze state at initialization. The haptics taught me that high-level APIs often call the simplest underlying system call, and sometimes you need to drop one level to get what you want.

These aren't React Native problems. They're abstraction problems. Every framework hides complexity until it doesn't, and the engineers who ship reliable software are the ones who know where the seams are. That's what building Arethos taught me — not how to use React Native, but how to debug it when the surface-level answer isn't enough.
